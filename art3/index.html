Google Cloud · Foundations

The guided foundation most teams skip

Before the first VM, the first Cloud Run service, the first bucket, there is a question almost nobody answers on purpose: how is this organization actually laid out? Google Cloud Setup answers it for you, and hands you the Terraform on the way out.

Knacketho Blogs Cloud Architecture ~11 min read

The one-line version

Google Cloud Setup is a console-guided flow that walks an org admin through ten foundational tasks (identity, IAM, billing, resource hierarchy, security, logging, and networking), then compiles those choices into Terraform you can deploy in one click or take into your own pipeline.

Most Google Cloud accounts begin the same way. Someone creates a project named my-first-project, enables a couple of APIs, and starts shipping. It works, right up until the org has forty projects, three of them production, nobody remembers who owns the billing account, and audit logs are scattered across every project instead of pooled somewhere you can actually query them.

The fix for that is a landing zone: a deliberate, opinionated baseline for how the whole organization is structured before workloads land on it. Building one by hand means reading a stack of architecture documents and stitching together org policies, IAM bindings, Shared VPC, log sinks, and folder hierarchies. Google Cloud Setup is the shortcut, an interactive flow inside the console that asks you the right questions, applies Google's recommended defaults, and emits the whole thing as deployable infrastructure-as-code.

It lives at console.cloud.google.com/cloud-setup, and it is genuinely underused. This is a walkthrough of what it does, when to reach for it, and where it stops.

01 / FoundationsPick a foundation that matches the workload

The flow opens by asking you to commit to one of three foundation options. This isn't cosmetic; your choice determines which of the downstream tasks you're guided through.

Lightweight

Proof of concept

Organization and billing only, plus a minimal folder + management + standard project. For experimenting before you commit.

Recommended

Production

The full ten-task flow, security, central logging, Shared VPC, hybrid connectivity. A secure, scalable baseline for a real org.

Regulated

Enhanced security

Everything in Production, plus Cloud KMS with Autokey enforcing customer-managed encryption keys across your environment folders.

If you're evaluating Google Cloud for the first time, Proof of concept gets you a clean, deployable structure in minutes without the networking and security depth. For anything you intend to run real traffic on, start with Production, and pick Enhanced security if you operate under strict compliance requirements where CMEK enforcement is non-negotiable.

02 / SequenceThe ten tasks, in order

The Production foundation is a checklist of ten tasks. They're sequenced deliberately: each one assumes the previous is done, because access to perform a later task depends on the groups and roles you created earlier.

01

Organization

Set up Cloud Identity or Google Workspace, verify your domain, and let the organization resource, the root of your hierarchy, get created automatically.

super administrator
02

Users & groups

Create the recommended admin groups (gcp-organization-admins, gcp-billing-admins, gcp-network-admins, and more) and put real people in them. Or federate Entra ID, Okta, or any OIDC/SAML provider.

super admin / org admin
03

Administrative access

Grant IAM roles to those groups at the organization level, never to individuals. The flow pre-fills sensible defaults you can trim.

org admin
04

Billing

Create or link a Cloud Billing account and attach it to the organization. Google recommends a single billing account per org.

gcp-billing-admins
05

Hierarchy & access

Choose a folder/project topology, environment-oriented, team-oriented, or business-unit-oriented, and set the IAM policies that flow down through it.

gcp-organization-admins
06

Security

Apply recommended org policies, enable Security Command Center (Standard tier is free), set Essential Contacts, and, on Enhanced security, turn on Cloud KMS Autokey.

org / security admins
07

Central logging & monitoring

Route org-wide audit logs into one log bucket with a chosen region and retention, optionally export via Pub/Sub, and put all new projects under one monitoring scope.

logging/monitoring admins
08

VPC networks

Stand up Shared VPC host projects for non-production and production, each with public and private subnets, firewall rules, Flow Logs, Private Google Access, and Cloud NAT.

gcp-network-admins
09

Hybrid connectivity

Build an HA VPN to your on-prem or other-cloud network, gateways, Cloud Router with BGP, and redundant tunnels. Upgrade to Cloud Interconnect later for something sturdier.

org admin
10

Deploy & support

Review every tab, then deploy directly from the console or download the Terraform. Finish by choosing a support plan.

gcp-organization-admins

Notice the pattern in the right-hand labels: the flow is teaching you role-based access control by making you live it. You can't run the billing task as the network admin, because the foundation assumes least privilege from the very first screen.

03 / HierarchyThe decision that's hard to undo

Of all ten tasks, the resource hierarchy is the one worth slowing down for. Folders and projects in Google Cloud inherit IAM policies and org policies top-down, so the shape you pick here becomes the shape of your access control, your billing attribution, and your blast radius for years.

Setup offers four starting templates:

  • Simple environment-oriented: isolate Non-production from Production. Good for a small company with centralized environments.
  • Simple team-oriented: isolate teams like Development and QA, with environment folders nested under each. Good for small companies with autonomous teams.
  • Environment-oriented: environments at the top, then business units, then teams. Good for large centralized orgs.
  • Business-unit-oriented: business units at the top, then teams, then environments. Good for large orgs where units operate independently.

Each template includes a Common folder for shared resources, exactly where your centralized logging and monitoring projects belong. The honest advice: model this on how your org actually delegates ownership and budget, not how the org chart looks on a slide. If billing and access naturally split by business unit, a team-first hierarchy will fight you forever.

Worth knowing It merges with what you already have. If your organization already contains folders and projects, the proposed hierarchy appears alongside them, and anything identical is merged automatically rather than duplicated. You don't have to start from an empty org to benefit.

04 / OutputIt's Terraform underneath

This is the part that earns Google Cloud Setup a place in a serious team's toolkit. Every choice you make across hierarchy, security, logging, networking, and hybrid connectivity is compiled into Terraform configuration files. On the final screen you choose how to apply them:

  • Deploy directly from the console: a one-time, fire-and-forget apply. Simple, and ideal for a proof of concept. You can only do this once.
  • Download and deploy the Terraform: pull down a terraform.tar.gz, commit it, and run it through your own workflow as many times as you like.
Decide before you click "Deploy directly" and "download" are a fork, not a sequence. If there's any chance you'll manage this foundation through your own Terraform pipeline later, do not deploy directly, take the download path from the start so the state lives where your team can iterate on it.

For anyone already running infrastructure-as-code, the download option is the obvious choice. You get a Google-blessed starting point (org policies, Shared VPC, log sinks, IAM) as code you own, version, and extend, instead of a black box you clicked into existence. A reasonable first move after download:

# unpack the generated foundation
tar -xzf terraform.tar.gz
cd cloud-setup

# review what it intends to create before anything runs
terraform init
terraform plan -out=foundation.tfplan

# commit it first, then apply through your normal pipeline
git add . && git commit -m "baseline gcp foundation from cloud setup"
terraform apply foundation.tfplan

05 / JudgementWhen to use it, and when not to

Google Cloud Setup is the right tool when you're standing up a new organization, when an existing org has grown organically and you want to retrofit a sane baseline, or when you need to learn what a well-architected foundation even looks like. The flow is a teaching instrument as much as a deployment one.

It's the wrong tool when you already have a mature, opinionated landing zone managed in code. At that point the enterprise foundations blueprint, or your own Terraform modules, give you finer control than a guided flow is designed to. Setup gets you to a strong default fast; it doesn't replace the deep customization a regulated enterprise eventually needs.

The realistic workflow for most teams sits in the middle: run the flow once to generate the foundation, choose download the Terraform, and treat the output as the first commit of an infrastructure repository you'll evolve from there. You get Google's accumulated opinions about hierarchy, IAM, logging, and networking as a starting line, and you keep full ownership of where it goes next.

The deeper you get into Google Cloud, the more the early, invisible decisions matter, the ones about structure that nobody sees until they're painful to change. Google Cloud Setup's real value isn't that it saves an afternoon of clicking. It's that it forces those decisions to the surface, in order, while they're still cheap to get right.
